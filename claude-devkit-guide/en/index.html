claude-devkit

A cross-project Claude Code plugin that goes deep on two high-frequency scenarios: PR scoring and planning-document composition. Built around progressive disclosure and switchable profiles — structural acknowledgment that there is no silver bullet.

1 · Install & Quick Start

Prerequisites

  • Claude Code CLI installed (official docs)
  • GitHub account authenticated via gh auth login (used by plan-writer's --pr and score-pr's diff reading)

Three lines to install

/plugin marketplace add suNmerandone/claude-devkit
/plugin install devkit@suNmerandone-claude-devkit
/devkit:score-pr

Default scope is user — plugin lives in ~/.claude/, works across every project, pollutes no repo.

Three scopes

Scope Command When to use
user (default) /plugin install <name> Personal use, across all projects
project /plugin install <name> --scope project Team-shared, written into .claude/settings.json
local /plugin install <name> --scope local This project only, gitignored, just for you

Update

/plugin marketplace update
/plugin update
/reload-plugins

2 · /devkit:score-pr — PR Quality Scoring

Evaluates the current branch against the base branch, outputs a grade A / B+ / B- / C / D plus a categorized action list.

Two ways to invoke

Interactive mode (recommended for newcomers — walks you through 4 questions):

/devkit:score-pr

Direct mode (power users — pass arguments, skip the prompts):

# All defaults
/devkit:score-pr

# Specify base branch
/devkit:score-pr develop

# Enable ensemble (3 independent evaluations, take the mode)
/devkit:score-pr --ensemble

# Pick a profile
/devkit:score-pr --profile brownfield

# Combined
/devkit:score-pr develop --ensemble --profile brownfield --lang en
Tip

At the end of interactive mode, the output appends the equivalent direct-mode command — copy it next time to skip the interactive flow.

Profile selection logic

All 5 profiles share the same scoring criteria and grade table — only the weights differ. Pick one via this decision tree:

Is this a legacy / brownfield project?
├─ Yes → brownfield
└─ No →
    Is it high-stakes (medical, finance, aviation, drivers)?
    ├─ Yes → safety-critical
    └─ No →
        MVP / POC stage?
        ├─ Yes → fast-iteration
        └─ No →
            Strict TDD culture + high coverage requirement?
            ├─ Yes → test-driven
            └─ No → default

Default vs Brownfield weights

The two most commonly swapped profiles. Full weights for the other three live in rubric.md.

default
Balanced · tests in place, not legacy
  • Correctness25%
  • Test coverage20%
  • Backward compat15%
  • Defensive coding15%
  • Maintainability10%
  • Change risk10%
  • Code quality5%
brownfield
Legacy · tests down, compat up
  • Correctness25%
  • Test coverage10%
  • Backward compat20%
  • Defensive coding15%
  • Maintainability10%
  • Change risk15%
  • Code quality5%
Note

The rubric author is upfront: these 5 weight sets are not industry-standard. SonarQube uses pass/fail gates; ISO 25010 doesn't prescribe weights at all. Pair this with team calibration — score 3–5 historical PRs, compare to how the team feels about them, adjust profile selection accordingly.

Three issue classes

Symbol Class Effect on grade
! Must fix Blocks merge; drops grade to B- or lower
~ Suggested Fixing lifts grade by 1 level
- Optional No grade impact

Ensemble mode

--ensemble spins up 3 independent subagents that evaluate in parallel, takes the mode as the final grade, and labels "consistency: high / medium".

  • Use it when: release review, single evaluation feels unstable, establishing a baseline for a new project
  • Cost: 3× runtime, higher token consumption
  • Day-to-day: single mode is fine

Project-specific rule ingestion

If the target project has .claude/CLAUDE.md, score-pr auto-loads it and treats these sections as supplementary rubric:

  • Coding Standards / Conventions
  • Brownfield Rules
  • High-Risk Zones / Legacy Surface
  • Architecture & DI

It also reads .claude/skill/SKILL.md to identify known high-risk zones. If either file is absent, the "Assumptions" section marks it as skipped.

3 · /devkit:plan-writer — Planning Document Composer

Synthesizes structured analysis / development / planning documents from multiple sources: tickets, Confluence pages, GitHub PRs, or manual descriptions. Supports interactive, hybrid, and direct modes.

Four data sources

Source Flag (repeatable) Dependency
Azure DevOps ticket --ticket AB#12345 Azure DevOps MCP
Jira ticket --ticket PROJ-123 Confluence MCP (includes Jira)
Confluence page --confluence <url-or-id> Confluence MCP
GitHub PR / branch --pr <url> / --branch <name> gh CLI
Manual description --desc "<text>" None
Design

plan-writer itself binds to no MCP server — it borrows whatever's already in your Claude Code environment. If a given MCP isn't installed, plan-writer flags that source as unavailable and continues synthesizing from the rest.

Three output templates

Template Structure When to use
generic Scope · Background · Requirements · Approach · Risks · Estimation General software projects
brownfield-plan + Document History · Brownfield Considerations · Confluence v2.1 ACs · 5.5h/day estimation Legacy-heavy projects or ones inheriting similar conventions
minimal Goal · Approach · Estimated Effort POC, proofs-of-concept, lightweight tasks

You can also pass --template <path> to any local markdown file as a custom template. If the project has .claude/plan-template.md, interactive mode will offer to use it.

Three execution modes

# Interactive mode — walks through 5–6 questions
/devkit:plan-writer

# Hybrid mode — some flags given, asks for the rest
/devkit:plan-writer --ticket AB#12345

# Direct mode — fully specified, silent and fast
/devkit:plan-writer --ticket AB#12345 --template brownfield-plan \
                   --output doc/plan/MyFeature.md

Behaviour on missing data

  • Sections with insufficient material get <TBD: what's missing> markers
  • All TBDs get consolidated into a Metadata section at the end of the doc
  • No fabrication of content that isn't in the source material
  • If every source fails → produces the template skeleton plus an explicit "info needed" list

What plan-writer does NOT do

Boundary

This is a planning tool, not an execution tool. It does not auto-create Jira subtasks, does not publish to Confluence, does not change ticket statuses, does not run any git operations (commit, push, branch switching).

Those belong to domain-specific plugins — if your team needs such workflows, build a separate plugin (e.g. acme-toolkit); don't mix them into devkit.

4 · Design Principles (these shape how you use it)

These aren't just philosophy — they actively shape where devkit works, where to stop using it, and what to fall back to.

SKILL.md vs README.md — separation of concerns

SKILL.md holds the English instructions for Claude (the model handles English system prompts better; output language is specified separately as Traditional Chinese). README.md is the human-facing doc in Traditional Chinese. The two do not duplicate each other, and neither annotates the other.

Progressive disclosure

SKILL.md is capped at 500 lines, focused on the core workflow. Detailed rules live in reference files (like rubric.md), which Claude loads on demand. This avoids diluting the main instruction and bloating token usage.

Stay generic · inject specialization via CLAUDE.md

devkit is positioned as a generic toolkit — no project-specific logic is hardcoded. When specialization is needed, it reads the target project's .claude/CLAUDE.md (already implemented in score-pr). Tooling tightly coupled to a single project belongs in that project's .claude/commands/, not in devkit.

Dual-mode + direct-mode hint

Every skill supports interactive, hybrid, and direct modes. At the end of interactive/hybrid runs, the output appends the equivalent direct-mode command — a deliberate learning curve: "try it once interactively, skip next time".

No built-in domain integrations

devkit does not ship MCP servers, write-back actions (e.g. auto-creating Jira subtasks), project-specific pre-commit hooks, or build wrappers. Those belong to domain plugins or project-level .claude/commands/.

5 · Extending & Contributing

Adding a skill

Full 11-step checklist in CONVENTIONS.md. Core skeleton:

  1. Create skills/<skill-name>/
  2. Write SKILL.md (English instructions, YAML frontmatter with description, output language set to Traditional Chinese)
  3. Write README.md (Traditional Chinese — purpose + usage + options + examples + design rationale)
  4. If detailed reference material exists → split into its own file (e.g. rubric.md)
  5. Templates go in templates/, samples go in examples/ (recommended)
  6. Update the root README's "Available tools" table + bump the plugin.json version (new skill = MINOR)

SemVer — three levels

Level Change type Example
PATCH Rubric tweaks, prompt wording, doc restructure Tweaking score-pr rubric weights
MINOR New skill, important new option Adding /devkit:plan-writer
MAJOR Output format or core dimension breaking change Changing score-pr grade thresholds

Non-Claude-Code users

For teammates using GitHub Copilot / Cursor / ChatGPT, standalone prompt versions exist:

  • prompts/score-pr.md
  • prompts/plan-writer.md

Same rubric, same output format — but no ensemble, no auto data collection, no project CLAUDE.md reading.

Takeaway

devkit doesn't chase "big and comprehensive" — it goes deep on two high-frequency scenarios: PR scoring and planning doc composition. Three structural choices are worth learning: separation of SKILL.md (machine-read) from README.md (human-read), using profiles to acknowledge there's no silver bullet, and injecting specialization by reading project CLAUDE.md rather than hardcoding it. A generic plugin that adapts to multiple engineering cultures, without forking a branch per customer.
