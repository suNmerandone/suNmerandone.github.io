claude-devkit

Claude Code 跨專案開發輔助 plugin，把 PR 評分與規劃文件合成兩個高頻場景做透。 走 Progressive Disclosure 與可切換 profile 的設計，用工具結構承認「沒有銀彈」。

1 · 安裝與快速上手

前置條件

  • Claude Code CLI 已安裝（官方文件）
  • GitHub 帳號已透過 gh auth login 登入（plan-writer 的 --pr 與 score-pr 的 diff 讀取會用到）

三行指令裝起來

/plugin marketplace add suNmerandone/claude-devkit
/plugin install devkit@suNmerandone-claude-devkit
/devkit:score-pr

預設是 user scope，plugin 寫到 ~/.claude/，所有專案都能用，不會污染任何專案的 repo。

三種 scope 差異

Scope 指令 適用
user（預設） /plugin install <name> 個人使用，跨所有專案
project /plugin install <name> --scope project 團隊共用，寫入 .claude/settings.json
local /plugin install <name> --scope local 僅此專案、gitignored、僅自己

更新

/plugin marketplace update
/plugin update
/reload-plugins

2 · /devkit:score-pr — PR 品質評分

評估當前 branch 相對於 base branch 的變更品質，輸出 A / B+ / B- / C / D 等級與分類行動清單。

兩種呼叫方式

互動模式（新手推薦，逐步詢問 4 個問題）：

/devkit:score-pr

快速模式（熟練者，帶參數直接執行）：

# 全部用預設
/devkit:score-pr

# 指定 base branch
/devkit:score-pr develop

# 啟用 ensemble（3 次獨立評估取眾數）
/devkit:score-pr --ensemble

# 指定 profile
/devkit:score-pr --profile brownfield

# 組合
/devkit:score-pr develop --ensemble --profile brownfield --lang en
Tip

互動模式結束時會在輸出末尾提示等價的 direct-mode 指令，下次可直接複製使用以跳過互動。

Profile 選擇邏輯

5 組 profile 的評分準則與等級對照完全相同，只有權重不同。依這個決策樹選：

是否為 legacy / brownfield 專案？
├─ 是 → brownfield
└─ 否 →
    是否為高風險領域（醫療、金融、航空、驅動）？
    ├─ 是 → safety-critical
    └─ 否 →
        是否為 MVP / POC 階段？
        ├─ 是 → fast-iteration
        └─ 否 →
            嚴格 TDD 文化 + 高測試覆蓋要求？
            ├─ 是 → test-driven
            └─ 否 → default

Default vs Brownfield 權重對照

最常被拿來切換的兩組。其他三組的完整權重見 rubric.md。

default
通用平衡 · 測試完善、非 legacy
  • 正確性25%
  • 測試覆蓋20%
  • 向後相容15%
  • 防禦性編碼15%
  • 可維護性10%
  • 變更風險10%
  • 程式碼品質5%
brownfield
大型 legacy · 測試壓低、相容提高
  • 正確性25%
  • 測試覆蓋10%
  • 向後相容20%
  • 防禦性編碼15%
  • 可維護性10%
  • 變更風險15%
  • 程式碼品質5%
Note

Rubric 作者誠實聲明：「這 5 組權重不是業界公認標準」。SonarQube 用 pass/fail 門檻、ISO 25010 不規定權重。建議搭配團隊校準使用 —— 選 3–5 個歷史 PR 跑評分，比對團隊感覺值，調整 profile 選擇。

三類問題符號

符號 分類 對等級的影響
! 必須修正 不處理則不應合併；壓低至 B- 或以下
~ 建議修正 修完可提升 1 級
- 可選改善 不影響等級判定

Ensemble 模式

--ensemble 啟動 3 個獨立 subagent 同時評估，取眾數為最終等級，標示「一致性：高 / 中」。

  • 用在：release 前審查、懷疑單次評估波動、新專案第一次建立基準
  • 代價：執行時間 3 倍、token 消耗增加
  • 日常：single mode 即可

專案特定規則自動讀取

若目標專案有 .claude/CLAUDE.md，score-pr 會自動讀取並套用以下章節作為補充 rubric：

  • Coding Standards / Conventions
  • Brownfield Rules
  • High-Risk Zones / Legacy Surface
  • Architecture & DI

也會讀 .claude/skill/SKILL.md 識別已知高風險區域。若專案無這些檔案，會在「評估假設」區段標示跳過。

3 · /devkit:plan-writer — 規劃文件合成

整合 Ticket / Confluence / GitHub PR / 手動描述 多方素材，合成結構化的分析/開發/規劃書。支援互動、混合、快速三種模式。

四種資料來源

來源 參數（可重複） 依賴
Azure DevOps Ticket --ticket AB#12345 Azure DevOps MCP
Jira Ticket --ticket PROJ-123 Confluence MCP（含 Jira）
Confluence 頁面 --confluence <url-or-id> Confluence MCP
GitHub PR / branch --pr <url> / --branch <name> gh CLI
手動描述 --desc "<text>" 無
Design

plan-writer 本身不綁定任何 MCP server —— 它借用你既有的 Claude Code 環境配置。若你沒裝某個 MCP，plan-writer 會標示該資料源不可用並繼續用其他資料合成。

三種輸出樣板

樣板 結構重點 適用
generic Scope · Background · Requirements · Approach · Risks · Estimation 一般軟體專案
brownfield-plan + Document History · Brownfield Considerations · Confluence v2.1 ACs · 5.5h/day 工時換算 大型 legacy 或延續類似慣例的專案
minimal Goal · Approach · Estimated Effort POC、概念驗證、輕量任務

也可用 --template <path> 指向任意本地 markdown 檔作為自訂樣板；專案若有 .claude/plan-template.md，互動模式會詢問是否使用。

三種執行模式

# 互動模式 — 逐步問 5–6 題
/devkit:plan-writer

# 混合模式 — 帶部分參數，缺的項目才問
/devkit:plan-writer --ticket AB#12345

# 快速模式 — 齊全參數、不問不吵
/devkit:plan-writer --ticket AB#12345 --template brownfield-plan \
                   --output doc/plan/MyFeature.md

資料不足時的行為

  • 素材不足的章節標記 <TBD: 具體缺什麼>
  • 所有 TBD 彙整到文件末的 Metadata 區段
  • 絕不捏造素材中沒有的資訊
  • 若所有資料源都失敗 → 產出樣板骨架 + 明確「需要補充的資訊」清單

plan-writer 不做什麼

Boundary

這是規劃工具，不是執行工具。不自動建立 Jira subtasks、不發佈 Confluence、不改 ticket 狀態、不執行任何 git 操作（commit / push / branch 切換）。

這些都屬於領域特定 plugin 的範疇 —— 若你的團隊需要這類 workflow，應建獨立 plugin（如 acme-toolkit），不該混入 devkit。

4 · 設計原則（影響你怎麼用）

這些原則不只是哲學，會實際影響你怎麼用、什麼時候該 fallback。

SKILL.md vs README.md 職責分離

SKILL.md 是給 Claude 讀的英文指令（AI 對英文 system prompt 反應較好、輸出語言另指定為繁中）。README.md 是給人讀的繁中使用者文件。兩者內容不重複、不互為註解。

Progressive Disclosure

SKILL.md 控制在 500 行內，聚焦核心工作流程。詳細規則拆到 reference 檔案（如 rubric.md），Claude 按需讀取。避免在主 prompt 塞入所有規則稀釋主要指令、增加 token 消耗。

保持通用 · 透過讀 CLAUDE.md 注入特化

devkit 的定位是通用工具，不硬編碼任何專案特定邏輯。需要專案特化時透過讀目標專案的 .claude/CLAUDE.md 注入規則（score-pr 已實作）。高度綁定特定專案的工具應放在該專案的 .claude/commands/，不進 devkit。

雙模式 + Direct-mode Hint

每個 skill 支援互動 + 混合 + 快速三種模式。互動或混合模式結束時會在輸出末尾提示等價的 direct-mode 指令，讓使用者能「試著用一次、下次直接跳過互動」。這是刻意設計的學習曲線。

不內建領域特定整合

devkit 不提供 MCP server、write-back 操作（自動建 Jira subtask 等）、專案特定 pre-commit hook 或 build wrapper。這些交給領域 plugin 或 .claude/commands/。

5 · 擴充與貢獻

新增 skill 的步驟

完整 11 步清單見 CONVENTIONS.md。核心骨架：

  1. 建 skills/<skill-name>/ 目錄
  2. 寫 SKILL.md（英文指令、YAML frontmatter 含 description，指定輸出為繁中）
  3. 寫 README.md（繁中，用途 + 使用 + 選項 + 範例 + 設計原理）
  4. 若有詳細參照資料 → 拆獨立檔（如 rubric.md）
  5. 樣板放 templates/、範例放 examples/（推薦）
  6. 更新根 README 的「可用工具」表格 + bump plugin.json 的 version（新 skill 為 MINOR）

SemVer 三層

層級 變更類型 範例
PATCH rubric 修正、prompt 措辭調整、文件重構 調整 score-pr rubric 權重
MINOR 新增 skill、新增重要選項 新增 /devkit:plan-writer
MAJOR 輸出格式或核心維度 breaking change 改變 score-pr 等級區間定義

非 Claude Code 使用者

團隊成員用 GitHub Copilot / Cursor / ChatGPT 時，可用獨立 prompt 版本：

  • prompts/score-pr.md
  • prompts/plan-writer.md

使用相同的 rubric 與輸出格式，但無 ensemble、無自動資料收集、無專案 CLAUDE.md 讀取。

Takeaway

devkit 不追求大而全，而是把 PR 評分與規劃書合成兩個高頻場景做透。三個結構性選擇值得學：SKILL.md（機器讀）與 README.md（人讀）職責分離、用 profile 承認「沒有銀彈」、用讀專案 CLAUDE.md 的機制注入特化而不硬編碼。讓一個通用 plugin 能適配多種工程文化，不需要為每個客戶派生分支。
