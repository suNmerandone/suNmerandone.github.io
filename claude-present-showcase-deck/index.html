Framing

今天想談的其實不是「AI 怎麼生內容」

那個問題已經被解到太多地方了。

我想談的是 —— 內容系統在 live 之後，怎麼自我治理。

這是 claude-present 從 v1 到 v2 的敘事轉向，也是這份 deck 存在的理由。

02 · Problem

v1 的病不是 v2 的病

v1 · 2026 上半年

Drift。
每次產出風格、結構、語氣都漂移。無法累積成資產。

v2 · 現在

Governance after live。
系統上線後，錯誤被發現之後怎麼辦？是人去救火，還是系統自己修、自己記、下次別人也不犯？

前者是「讓 AI 能產」的問題，後者是「讓 AI 產的系統能活」的問題。

03 · Foundation · 來自 v1 的三原則

三層結構，限制 AI 自由度

  • 01 Design system 單一真實來源 tokens.json 是所有視覺的唯一來源，任何 skill 不能自己發明顏色或字型。
  • 02 Skills 必讀 gold standard 每個 skill 觸發時必須先讀 examples/，禁止憑空生成；讓過去的好產出 anchor 未來。
  • 03 Outputs 三件組 每次產出都帶 index.html + source-notes.md + generation-log.md，讓產出同時是資料 + metadata + 追溯訊號。
04 · Architecture

三層架構的依賴方向

05 · Exhibit A · Governance

銘傳譯名事件 · 系統自己發現、自己修

Situation

resume-profile skill 首次用於本人履歷時，AI 將銘傳大學 Information and Telecommunications Engineering 誤譯為「資訊傳播工程」—— 正確譯名應為「電腦與通訊工程」（依銘傳大學官方系網 sites.google.com/view/mcuite 為準）。

Action

使用者 review 時抓出、指認為「非常嚴重」（履歷的譯名錯誤直接衝擊信任）。當下：

  • 修：resume.html 的譯名當下修正
  • 記：把教訓編碼進 .claude/skills/resume-profile/references/anti-patterns.md 的第 5 條鐵則
  • 鎖：commit 8a1fc3a 的 body 留完整紀錄
05 · Exhibit A · Result

一次錯誤 → 一條鐵則 → 同類錯再也不犯

.claude/skills/resume-profile/references/anti-patterns.md# 鐵則 5 · 專有名詞翻譯必須 Google 雙檢 # 新增於 2026-04-18（銘傳譯名事件） 學校 / 系所 / 公司 / 產品的官方譯名： - 不准自己推理 - 必須 Google 雙檢 - 若找不到官方譯名，標註「未驗證」而非自信輸出 # 理由：履歷的譯名錯誤直接衝擊信任， # 這不是格式問題是可信度問題。

這條規則從這一刻起，任何 resume-profile 產出都會遵守。這就是 governance loop：錯誤不只被修，還被系統記得。

06 · Exhibit B · Self-publish

私 → 公單向管道的三層鎖

Situation

個人 AI 產出天然需要分 public（可公開）和 private（email、未驗證數字、內部事件）。但手動切兩份會漂移 — 要一套機制「一份 source，兩份產出」。

Action

設計 promote-to-showcase skill，三層鎖：

outputs/resume/v0.2.0_.../resume.html ├─ <div class="private-only">...</div> ← 層 1：私用 marker ├─ 「電話：0912-xxx-xxx」 ← 層 2：欄位過濾（email/tel 等） └─ <!-- sync-marker: ZH-... --> ← 層 3：internal comment 移除 public-showcase/resume/index.html ← 自動重生，永遠不可能帶回私內容
06 · Exhibit B · Result

一個 skill 啟動，兩個 repo 同步完成

  • 單向：只做私 → 公，絕不反向 pull back（避免公開端污染私源頭）
  • 顯式：每次要人**明確指名**哪個 output，絕不自動 promote 全部
  • 可追：commit + push 都在 destination repo 留紀錄

9ed8654 skill 本身 · b3a4f72 landing markers 改造 · f487378 首次 promote 驗證

能直接套用的原則：凡是要分公開／私用的系統，把 filter 做在發布管道而不是 source。Source 保持完整，發布端做節制 — 這比「維護兩份 source」穩定太多。

07 · Exhibit C · Identity

個人身份 · 信任成本壓力測試

Situation

resume / LinkedIn 是任何內容系統最嚴苛的壓力測試 — 訊息一旦錯誤，信任成本是實質的職涯傷害，不是重跑一次就解決。

Action

為 resume-profile skill 加上三層支撐：

  • inputs/resume/linkedin/ 放 PDF snapshot 作為 LinkedIn 現況基準
  • LinkedIn refresh / rebrand 雙模式指南（例行維護 vs 策略重塑）
  • interview-material 保留機制：訪談原始資料永遠不刪，歷次履歷都能追溯回原話
07 · Exhibit C · Result

支援「延後執行」 · 信任敏感系統的共通需求

使用者明確要求「延後執行 LinkedIn rebrand」時，系統不強迫現在做完。

resume-profile · Step 6 · 詢問是否產出 LinkedIn 更新指南 refresh — 例行維護（微調文字、補新專案） rebrand — 策略性重塑（重新定位個人品牌） skip — 本次不需要（可於日後再觸發，LinkedIn snapshot 仍保留）

deferred execution 是任何處理「信任敏感」內容的系統都該內建的機制：讓使用者有決定「現在不做」的權利，而不是被系統推著跑。

Takeaway
Maturity 不是 output 量 ——
是 governance loops。
  • 01 — 系統能產多少，不如系統自己發現錯誤後能記得多少
  • 02 — 編碼規則比產出規則更值錢，因為前者讓未來不再犯同類錯
