Framing

What I actually want to talk about isn't "how AI generates content"

That problem has been solved in too many places already.

I want to talk about how a content system governs itself — after it's live.

This is claude-present's narrative pivot from v1 to v2, and the reason this deck exists.

02 · Problem

v1's disease isn't v2's disease

v1 · early 2026

Drift.
Every output drifts in style, structure, tone. Nothing compounds into an asset.

v2 · now

Governance after live.
Once the system is live, what happens after an error is found? Does a human firefight, or does the system fix itself, remember, and never let anyone hit the same class of error again?

The former is "can the AI produce?" The latter is "can the system producing the AI survive?"

03 · Foundation · inherited from v1

Three-layer constraint on AI's freedom

  • 01 Design system as single source of truth tokens.json is the only source of visual decisions. No skill invents its own colors or fonts.
  • 02 Skills must read a gold standard Every skill reads examples/ before producing. No blank-page generation — past quality anchors future output.
  • 03 Three-file output set Every output ships with index.html + source-notes.md + generation-log.md — the output is data + metadata + traceability.

05 · Exhibit A · Governance

Mingchuan translation incident · system noticed, system fixed

Situation

The very first time resume-profile skill was used on the user's actual résumé, the AI rendered Mingchuan University's Information and Telecommunications Engineering as "資訊傳播工程" — the correct Chinese is "電腦與通訊工程" (per the official Mingchuan CCE department site, sites.google.com/view/mcuite). A plausible-sounding but wrong rendering.

Action

The user caught it on review and flagged it as "very serious" — a mistranslation on a résumé directly hits credibility. In the moment:

  • Fix: correct the translation in resume.html immediately
  • Remember: encode the lesson as rule 5 in .claude/skills/resume-profile/references/anti-patterns.md
  • Lock: commit 8a1fc3a body carries the full record
05 · Exhibit A · Result

One error → one rule → same failure class cannot recur

.claude/skills/resume-profile/references/anti-patterns.md# Rule 5 · Proper nouns must be Google-verified # Added 2026-04-18 (Mingchuan incident) For institutions / departments / companies / products: - Do NOT infer translations - MUST Google-verify the official rendering - If no official translation exists, mark "unverified" instead of outputting with false confidence # Reason: a mistranslation on a résumé is not a formatting # issue — it's a credibility issue.

From that moment on, every resume-profile output obeys this rule. That's a governance loop: the error isn't just fixed, it's remembered by the system.

06 · Exhibit B · Self-publish

Private → public one-way pipeline · three filter layers

Situation

Personal AI outputs naturally need to split into public (OK to share) and private (email, unverified numbers, internal events). But hand-maintaining two copies drifts — we need one source, two destinations.

Action

Built the promote-to-showcase skill with three filter layers:

outputs/resume/v0.2.0_.../resume.html ├─ <div class="private-only">...</div> ← layer 1: private marker ├─ "Tel: 0912-xxx-xxx" ← layer 2: field filter (email/tel/etc) └─ <!-- sync-marker: ZH-... --> ← layer 3: internal comments stripped public-showcase/resume/index.html ← auto-regenerated, can never carry private back
06 · Exhibit B · Result

One skill invocation · two repos synchronized

  • One-way: private → public only. Never pull back (so public never contaminates the private source).
  • Explicit: user must name the output to publish. No blanket auto-publish of everything.
  • Traceable: commits + pushes logged in the destination repo.

9ed8654 skill itself · b3a4f72 landing marker redesign · f487378 first promote validation

Portable principle: when you need to split public / private, put the filter on the publishing pipeline, not the source. Keep source complete, be austere at the edge — far more stable than maintaining two sources.

07 · Exhibit C · Identity

Personal identity · the trust-cost stress test

Situation

Résumé / LinkedIn is the harshest stress test any content system will face — wrong info doesn't just mean a re-run. The trust cost is real career damage.

Action

Added three supports to resume-profile:

  • inputs/resume/linkedin/ stores a PDF snapshot as LinkedIn ground-truth baseline
  • Dual LinkedIn refresh / rebrand guides (routine maintenance vs strategic repositioning)
  • interview-material preservation: raw interview data never deleted, every résumé iteration traceable back to source
07 · Exhibit C · Result

Deferred execution · a must-have for trust-sensitive systems

When the user said "defer the LinkedIn rebrand", the system didn't force completion now.

resume-profile · Step 6 · Ask if LinkedIn guide is needed refresh — routine maintenance (polish text, add new projects) rebrand — strategic repositioning (overhaul personal brand) skip — not this time (can re-trigger later; LinkedIn snapshot stays)

Deferred execution should be built into any system touching trust-sensitive content: give the user the right to say "not now" instead of being shoved forward by the system.

Takeaway
Maturity isn't output count —
it's governance loops.
  • 01 — How much a system produces matters less than how much it remembers after an error.
  • 02 — Encoding rules is worth more than producing outputs — the former keeps future runs from hitting the same class of failure.
  • 03 — A system that can publish itself will outlive one that waits for humans to update it — by about a decade.
