Framing

不追求大而全

兩個 skill（/devkit:score-pr、/devkit:plan-writer）加起來支撐的是日常工程兩個最高頻的決策點。

用工具結構**承認「沒有銀彈」** —— 不同工程文化用不同 profile，rubric 準則相同但權重切換。

02 · 安裝

三行指令裝起來 — 預設 user scope

# Claude Code CLI 已裝、gh auth login 已登入 /plugin marketplace add suNmerandone/claude-devkit /plugin install devkit@suNmerandone-claude-devkit /devkit:score-pr

預設寫入 ~/.claude/，所有專案通用，**不污染任何 repo**。

若要團隊共用 → --scope project（寫入 .claude/settings.json）；若要僅此專案私用 → --scope local（gitignored）。

03 · 更新 / Scope 切換

更新跟切換的 3 行

# 更新： /plugin marketplace update /plugin update /reload-plugins

Scope 快速選擇

  • user（預設）— 個人使用、跨所有專案
  • project — 團隊共用、寫入 .claude/settings.json
  • local — 僅此專案、gitignored、僅自己
Section II · Tool A

/devkit:score-pr
PR 品質評分

等級 A / B+ / B- / C / D  ·  分類行動清單
04 · score-pr · 概觀

兩種呼叫：互動 vs 快速

# 互動模式 — 新手推薦，逐步問 4 題 /devkit:score-pr # 快速模式 — 熟練者、帶參數 /devkit:score-pr develop --ensemble --profile brownfield --lang en

互動模式結束會在輸出末尾**提示等價的 direct-mode 指令** — 下次直接複製使用，刻意的學習曲線設計。

05 · Profile 決策樹

5 組 profile 的選擇邏輯

是否為 legacy / brownfield 專案？ ├─ 是 → brownfield └─ 否 → 是否為高風險領域（醫療 / 金融 / 航空 / 驅動）？ ├─ 是 → safety-critical └─ 否 → 是否為 MVP / POC 階段？ ├─ 是 → fast-iteration └─ 否 → 嚴格 TDD 文化 + 高測試覆蓋？ ├─ 是 → test-driven └─ 否 → default

所有 profile 的**評分準則與等級對照完全相同**，只有 7 維度權重不同。

06 · Default vs Brownfield

最常被切換的兩組權重

default
通用平衡 · 測試完善、非 legacy
  • 正確性25%
  • 測試覆蓋20%
  • 向後相容15%
  • 防禦性編碼15%
  • 可維護性10%
  • 變更風險10%
  • 程式碼品質5%
brownfield
大型 legacy · 測試壓低、相容提高
  • 正確性25%
  • 測試覆蓋10%
  • 向後相容20%
  • 防禦性編碼15%
  • 可維護性10%
  • 變更風險15%
  • 程式碼品質5%
07 · Ensemble mode

把 LLM 評估的隨機性
用統計層補償

--ensemble 啟動 **3 個獨立 subagent 同時評估**，取眾數為最終等級，標示「一致性：高 / 中」。

  • 用在：release 前審查 · 懷疑單次評估波動 · 新專案第一次建立基準
  • 代價：執行時間 3 倍、token 消耗增加
  • 日常：single mode 即可

把 ML 的集成學習（ensemble learning）概念搬到日常工程工具 —— 這種「跨領域類比」是 senior → staff engineer 的思維分水嶺。

Section III · Tool B

/devkit:plan-writer
規劃文件合成

Ticket  ·  Confluence  ·  PR  ·  手動描述  → 結構化 plan
08 · plan-writer · 四種資料源

四個來源可混用

  • Azure DevOps Ticket --ticket AB#12345 Azure DevOps MCP
  • Jira Ticket --ticket PROJ-123 Confluence MCP（含 Jira）
  • Confluence 頁面 --confluence <url> Confluence MCP
  • GitHub PR / branch --pr <url> gh CLI
  • 手動描述 --desc "..." 無依賴

plan-writer 本身不綁定任何 MCP — 借用你既有的 Claude Code 環境配置。少哪個就標示不可用、用其他資料繼續合成。

09 · Templates · Modes

三樣板 × 三模式

  • generic — Scope / Background / Requirements / Approach / Risks / Estimation
  • brownfield-plan — 加 Document History、Brownfield Considerations、5.5h/day 工時換算
  • minimal — Goal / Approach / Estimated Effort（POC 用）
# 互動模式 — 逐步問 5-6 題 /devkit:plan-writer # 混合模式 — 帶部分參數，缺的才問 /devkit:plan-writer --ticket AB#12345 # 快速模式 — 齊全參數、不問不吵 /devkit:plan-writer --ticket AB#12345 --template brownfield-plan \ --output doc/plan/MyFeature.md
10 · Boundary · 不做什麼

這是規劃工具、不是執行工具

不自動建 Jira subtasks · 不發佈 Confluence · 不改 ticket 狀態 · 不執行任何 git 操作（commit / push / branch 切換）

這些屬於**領域特定 plugin** 的範疇 —— 若團隊需要這類 workflow，應建獨立 plugin（如 acme-toolkit），不混入 devkit。

資料不足時的行為

  • 素材不足的章節標記 <TBD: 具體缺什麼>
  • TBD 彙整到文件末的 Metadata 區段
  • 絕不捏造素材中沒有的資訊

11 · 四個結構性選擇

知道這些才能把工具用到位

  • SKILL.md vs README.md 職責分離 — SKILL.md 英文（給 Claude）、README.md 繁中（給人）；兩者不重複不互注
  • Progressive Disclosure — SKILL.md ≤ 500 行，詳細規則拆 reference 檔（如 rubric.md）按需讀，不稀釋主指令
  • 通用 + 讀 CLAUDE.md 注入特化 — devkit 不硬編碼任何專案邏輯；需要特化時讀目標專案的 .claude/CLAUDE.md 注入（score-pr 已實作）
  • Direct-mode Hint — 互動 / 混合模式結束時提示**等價的 direct-mode 指令**，讓使用者「試一次 → 下次直接跳過互動」
Takeaway
不追求大而全 ——
把**兩個高頻場景**做透
再用 profile 承認沒有銀彈。
  • 01 — SKILL.md（機器讀）與 README.md（人讀）職責分離，比「一份到底」更能擴展
  • 02 — 用 profile 切換權重，比寫死 rubric 更能適配多工程文化
  • 03 — 讀 CLAUDE.md 注入特化，讓通用 plugin 適配多專案，不需為每家客戶派生分支
