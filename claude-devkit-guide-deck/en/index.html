claude-devkit

A Claude Code cross-project plugin that takes two high-frequency scenarios all the way — PR scoring and plan-document synthesis.
v0.4.2 · MIT · sunmerandone/claude-devkit 2026-04-19
Framing

Not trying to be everything

Two skills (/devkit:score-pr, /devkit:plan-writer) — both aimed at the decision points developers hit most often.

The structure itself concedes "there is no silver bullet" — one rubric, five swappable weight profiles to match different engineering cultures.

02 · Install

Three commands · user scope by default

# Prereq: Claude Code CLI installed, gh auth login done /plugin marketplace add suNmerandone/claude-devkit /plugin install devkit@suNmerandone-claude-devkit /devkit:score-pr

Writes to ~/.claude/ — available in every project, contaminates no repo.

For team-wide use → --scope project (writes to .claude/settings.json). For project-local private → --scope local (gitignored).

03 · Update · Scope

Three lines to update & scope decisions

# Update: /plugin marketplace update /plugin update /reload-plugins

Quick scope picker

  • user (default) — personal use, across all projects
  • project — team shared, written to .claude/settings.json
  • local — this project only, gitignored, self only
Section II · Tool A

/devkit:score-pr
PR quality grading

A / B+ / B- / C / D  ·  categorized action list
04 · score-pr · overview

Two invocation modes: interactive vs direct

# Interactive — recommended for new users, asks 4 questions /devkit:score-pr # Direct — with flags /devkit:score-pr develop --ensemble --profile brownfield --lang en

Interactive runs end with a print of the equivalent direct-mode command — copy it next time to skip the interaction. A deliberate learning curve design.

05 · Profile decision tree

How to pick between 5 profiles

Legacy / brownfield project? ├─ Yes → brownfield └─ No → High-risk domain (medical / finance / aviation / drivers)? ├─ Yes → safety-critical └─ No → MVP / POC stage? ├─ Yes → fast-iteration └─ No → Strict TDD culture, high coverage required? ├─ Yes → test-driven └─ No → default

All profiles use identical scoring criteria and grade thresholds — only the 7-dimension weighting differs.

06 · Default vs Brownfield

The two most-swapped weightings

default
balanced · mature tests, non-legacy
  • Correctness25%
  • Test coverage20%
  • Backward compat15%
  • Defensive code15%
  • Maintainability10%
  • Change risk10%
  • Code quality5%
brownfield
legacy-heavy · tests down, compat up
  • Correctness25%
  • Test coverage10%
  • Backward compat20%
  • Defensive code15%
  • Maintainability10%
  • Change risk15%
  • Code quality5%
07 · Ensemble mode

Offsetting LLM randomness
with a statistical layer

--ensemble fires 3 independent subagents, takes the mode as the final grade, and labels "agreement: high / medium".

  • When: release review · suspecting single-run variance · first-time baseline on a new project
  • Cost: 3× execution time, higher token consumption
  • Daily: single mode is fine

Applying ensemble learning (ML concept) to a daily engineering tool. This kind of cross-domain analogy is the senior → staff watershed.

Section III · Tool B

/devkit:plan-writer
Plan synthesis

Ticket  ·  Confluence  ·  PR  ·  manual → structured plan
08 · plan-writer · four sources

Four sources, mixable

  • Azure DevOps Ticket --ticket AB#12345 Azure DevOps MCP
  • Jira Ticket --ticket PROJ-123 Confluence MCP (inc. Jira)
  • Confluence page --confluence <url> Confluence MCP
  • GitHub PR / branch --pr <url> gh CLI
  • Manual description --desc "..." no dependency

plan-writer itself binds to no MCP — it reuses your existing Claude Code environment. Missing MCPs are flagged "unavailable"; the plan composes from what's available.

09 · Templates · Modes

Three templates × three modes

  • generic — Scope / Background / Requirements / Approach / Risks / Estimation
  • brownfield-plan — adds Document History, Brownfield Considerations, 5.5h/day effort conversion
  • minimal — Goal / Approach / Estimated Effort (for POCs)
# Interactive — asks 5-6 questions /devkit:plan-writer # Hybrid — partial flags, asks only what's missing /devkit:plan-writer --ticket AB#12345 # Direct — full flags, silent /devkit:plan-writer --ticket AB#12345 --template brownfield-plan \ --output doc/plan/MyFeature.md
10 · Boundary · what it won't do

Planning tool, not execution tool

Won't auto-create Jira subtasks · won't publish to Confluence · won't change ticket status · won't run any git operation (commit / push / branch switch).

These belong to domain-specific plugins — if your team needs that workflow, build a separate plugin (e.g. acme-toolkit) rather than mixing it into devkit.

Behaviour under missing data

  • Sections lacking material marked <TBD: what's missing>
  • TBDs aggregated into a Metadata block at the file's end
  • Never fabricates what isn't in the sources
Section IV · Design principles

What
shapes usage

SKILL.md / README.md split  ·  Progressive Disclosure  ·  CLAUDE.md injection
11 · Four structural choices

Worth understanding to use the tool well

  • SKILL.md vs README.md split — SKILL.md is English (for Claude); README.md is Chinese (for humans). The two do not duplicate and do not cross-reference each other.
  • Progressive Disclosure — SKILL.md ≤ 500 lines; detailed rules pulled into reference files (e.g. rubric.md) loaded on demand to avoid diluting the main prompt.
  • Generic + CLAUDE.md injection — devkit hardcodes no project logic. Project specialisation is injected by reading the target repo's .claude/CLAUDE.md (score-pr implements this).
  • Direct-mode hint — interactive and hybrid modes print the equivalent direct-mode command at the end; users learn by doing once, then skip interaction next time.
Takeaway
Not trying to be everything —
take two high-frequency scenarios all the way,
then use profiles to admit
there's no silver bullet.
  • 01 — SKILL.md (machine-read) + README.md (human-read) split scales better than "one doc fits both"
  • 02 — Swappable profiles beat a hardcoded rubric for fitting multiple engineering cultures
  • 03 — Reading CLAUDE.md to inject specialisation lets a generic plugin adapt per-project without forking
