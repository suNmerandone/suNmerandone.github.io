02 · Three frictions

The resume-profile skill asks for three prerequisites

Before · using the skill
  • Install Node · install Claude Code · sign in
  • Understand the .claude/skills/ layout
  • Hand-edit Markdown to provide inputs
  • Session dies mid-flow → lose everything
After · using Presentkit
  • Double-click Presentkit.App.exe
  • Wizard forms · auto-saved drafts
  • Paste API key · see LIVE badge before spending
  • Press Generate · can cancel · can iterate

The former is a tool for "making AI produce"; the latter is a product non-developers can actually adopt.

04 · Data flow

One Generate run · what happens

GenerateViewModel.RunAsync()WizardVMs → ResumeIntake record → PromptAssembler.Build(intake, design, linkedIn, targetLang) → IClaudeClient.StreamMessageAsync # mock OR real by factory → StreamDelta accumulated into StringBuilder → ResumeOrchestrator.BuildArtifact # wrap HTML skeleton → ResumeFileWriter.Write # 4-file set → auto-advance → Step 8 (Done) # Bilingual mode loops twice with distinct LanguageDirective # If 2× per-call estimate > 25K, insert silent pause between calls

The whole flow has exactly one path to a real client: ClaudeClientFactory.Create. Going around it defeats offline mode, so CLAUDE.md lists it as a hard rule.

Section III · The 8-step wizard

Walk it
from the user's side

Design · LinkedIn · Intake · Materials · Anchor · Outline · Generate · Done
05 · Step 1
Design

Pick one, or bring your own

  • Tech-forward Coral (default) · software / AI / product
  • Corporate Neutral · finance / enterprise · deep blue, zero ornament
  • Minimalist Mono · design / editorial · all IBM Plex Mono
Or Import custom design (JSON) to bring your own tokens — accent and font stacks are all overridable.
06 · Step 2
LinkedIn · optional

Drop a PDF · 100% local

  • PdfPig extracts text → regex parser splits fields
  • If Intake is empty, silently auto-fills
  • If Intake has content → asks "Overwrite?" first
  • Zero tokens · all local parsing
Placed before Intake on purpose. One PDF fills seven form sections; the user only needs to proof-read.
07 · Step 3

Intake · seven sections, no more no less

1 · Name+Headline
The one-line positioning · sets the resume's voice
2 · Contacts
Free text — email / phone / LinkedIn / personal site, one per line
3 · Target role
The single most important field · drives all content selection
4 · Current position
Company · Role · Period · bullets (use numbers where available)
5 · Past experiences
Repeater · reverse-chronological · 10+ years old can be merged
6 · Education
School · degree · year
7 · Key achievements
3–5 items the reader would remember after an 8-second scan

Output language moved from Intake to the Generate step — because the 2× cost decision should happen at the moment of spend.

08 · Step 4
Materials · optional

Supplementary files · .pdf / .md / .txt

  • Job descriptions (Claude can tune the tone)
  • Old resumes / cover-letter drafts
  • Project READMEs · autobiographical notes
No tokens at this step; files ride along in the Generate prompt (adds a few thousand tokens, a few cents extra).
09 · Step 5
Anchor · optional

Mirror · or Iterate

  • Mirror style · inherit voice, regenerate content
  • Iterate (diff) · prior version as content baseline, apply only the change request
  • Iterate produces a machine-readable <!-- CHANGES --> log
Anchor only accepts Presentkit outputs (detects <!-- generated-by: presentkit -->). Arbitrary HTML is rejected — use Materials for that.
10 · Step 7

Generate · three gates before spending

InfoBar
Cost advisory: mode, estimated cost range, token estimate. < 20K green · 20–28K yellow · ≥ 28K red + actionable trim guidance
Radio
Output language: Chinese (1 call) / English (1 call) / Both (2 calls · ≈2× cost). Decision happens at the moment of spend
Pre-flight
Confirm dialog pops on click, showing model + cost + token + rate-limit risk. Cancel is the default button · skipped in mock mode

Step 6 Outline is a static review gate — the assembled Intake + Anchor is shown for one last confirmation · no API call · no cost.

11 · Step 8
Done

Four files · one complete artefact

  • resume.html · primary output · self-contained HTML
  • resume.en.html · generated only in Both mode
  • source-notes.md · inputs captured for reproducibility
  • generation.log.md · metadata + CHANGES block on iterate
Three buttons: open HTML, open folder, start a new run. Deliberate: files stay, wizard resets.
Section IV · BYOK safeguards

Trust
as a feature

use own key · see own spend · reconcile own billing
12 · Mode badge

Three-state pill in the top bar · always visible

LIVE

Amber · spends tokens

Real API key configured · Generate will charge · deliberately NOT red (reads as error)

OFFLINE

Slate · safe mode

MockClaudeClient · zero API calls · zero cost · deliberately NOT green (reads as "go")

NO KEY

Zinc · not configured

No key · Generate will fail · hints user to Settings: paste a key or flip OFFLINE

Refreshes whenever the Settings dialog closes — user always knows which mode they're in.

13 · Pre-flight + audit

Confirm before Generate · record after Generate

Pre-flight confirm dialog
Shows model, cost range, token estimate, rate-limit risk flag. Cancel is the default button — user must explicitly agree. Mock mode skips the dialog entirely.
Token estimate · 3 tiers
< 20K green · 20–28K yellow · ≥ 28K red + guidance ("remove Anchor / trim Materials / skip LinkedIn").
API call audit log
Every real call writes a line to %LOCALAPPDATA%\Presentkit\api-calls.log (tab-separated: timestamp / model / kind / tokens). Mock calls never touch it.
Stale-pref protection
Every real-API path re-reads prefs from disk. An early version cached, causing Generate to silently hit the real API after user flipped OFFLINE — this rule kills that class of regression.

This audit log lines up against Anthropic Console billing — "what did I actually spend" is always verifiable in under 10 seconds.
